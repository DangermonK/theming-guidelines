Search Results Should Use a ‘Load More’ Approach #381

Issue: Endless scrolling encourages more extensive scrolling, removing focus from the best-matching results.

Advice: Search results lists should use a “Load more” button for loading additional products, ideally with a dynamic number of products loaded by default.

Issue first observed Jun 2014. Guideline updated 4 times since, most recently Feb 2023.

Search Results Should Use a ‘Load More’ Approach #381

Mark as Read

Importance:Detail

Platform:Desktop Web

Research Findings

In general, sites shouldn’t just load the entire product list all at once when there are more than ~100 products, due to performance issues, data costs, and a potentially unreachable footer. Most e-commerce sites therefore rely on one of three primary ways for navigating and loading additional products of a larger product list: 1) pagination links, 2) a “Load more” button, and 3) endless scrolling.​‌‌‌‍‌‌‌‌‌‌‍‌‌‌‍‍‌‌‌‌‌‌‎

Pagination: Many participants used the number of pagination links to gauge how many results there were in total (see #377). While pagination links offered more control in terms of jumping to a specific “page”, very few participants actually used these during testing. Instead, they almost exclusively went to the “next” or “previous” page in the sequence.

We’ve identified another 82 examples of ‘pagination links for search results,’ illustrating this implementation type across 81 case studies and 12 different industries.

View 82 Desktop Web examples

1) Pagination links – A set of links typically located at the top and bottom of the product list. The total product list is broken into a number of “pages” of products, with each pagination link representing its own page. When a link is clicked, the current page of products is replaced with the products from the selected page. While the load may be via AJAX it’s most often just a regular page load.

“Load more”: Only a few of the tested sites used a “Load more” button for fetching additional product list results, but the participants responded very well to the implementation. The user interface is incredibly simple and all the user has to decide is whether they want to load more results and then click a button if they do.

We’ve identified another 29 examples of ‘’Load More’ button for search results,’ illustrating this implementation type across 29 case studies and 13 different industries.

View 29 Desktop Web examples

2) “Load more” button – A single button located at the end of the product list. When the “Load more” button is clicked, the next set of results is appended to the current product list (loaded in via AJAX). While similar to pagination links, it is a simpler user interface with just a single button as opposed to a set of links. Also, new results are appended to the current list as opposed to replacing it, which allows the user to scroll a larger set of results – potentially the entire list of results if the user loads in all of them.

Endless scrolling: Participants would often browse a lot more products when new results were loaded with endless scrolling. This meant more of the total results list was viewed; however, the first set of results consequently got less attention when compared to pagination and “Load more” implementations.

We’ve identified another 9 examples of ‘endless scrolling for search results,’ illustrating this implementation type across 9 case studies and 5 different industries.

View 9 Desktop Web examples

3) Endless scrolling – There’s no UI. Results are instead appended to the current product list by monitoring the user’s scrolling position and then loading in the next set of results (via AJAX) as the user nears the bottom of the product list. It’s a technically more complex solution than the pagination and “Load more” implementations and requires solid coding and UX sensibilities to be implemented properly to avoid sudden page jerks and to ensure that results are loaded in at the appropriate time.

The Pros & Cons for Search Results Lists

Throughout testing, numerous participants explicitly complained about pagination. They generally perceived pagination to be slow, and seeing more than a handful of pagination links would often discourage them from browsing the list. In general, the participants tended to browse much less of the total product list than on sites that relied on a “Load more” button or endless scrolling. On the upside, the participants spent relatively more time on the first page of results.

Notice the length of the scrollbar. On sites with endless scrolling, the test participants often browsed 100+ products, which they almost never did on pages with pagination and only rarely did on pages with a “Load more” button. While this proved efficient for the first 50 to 150 products, some participants would just continue scrolling the product list without really focusing on the individual products if there wasn’t a natural break in the product list. Thus, the initial benefit of endless scrolling would turn into an issue.

With endless scrolling, the user largely experiences the page as if the entire product list was loaded at once but without the performance penalty of loading in potentially hundreds of products on each search regardless of whether they will be seen or not. It therefore comes as no surprise that the participants browsed vastly more products on sites with endless scrolling. However, the initial results received comparatively less exposure. Endless scrolling is therefore ideal for quickly seeing the breadth of an entire category, but because users aren’t naturally halted when scrolling, they tend to scan more and focus less on individual items.

Due to the open nature of search, it tends to have far more results than browsing categories. Hundreds of search results are not an uncommon sight and, at mass merchant sites, users’ queries often return thousands of results. Furthermore, as search results are by default sorted by their relevance to the search query, there is less of a need for scrolling before getting relevant results (compared to category navigation) – indeed, the more the user scrolls, the less relevant the items will be. It’s therefore unwise to encourage users to scroll large quantities of search results, and therefore endless scrolling shouldn’t be used for the search results list. Pagination or a “load more” button will be better for search results as they don’t encourage superficial browsing of a large number of products, but instead nudge the user to focus more on exploring the first “page” of results.

With “Load more”, here seen at Rakuten, users get a natural break as the relevancy of the results decreases (unlike endless scrolling) but still have the option to compare items on “page 1” directly to items on “page 2” (unlike pagination). Ideally the threshold for how many products are loaded into the default results page dynamically changes based on the relevancy score drop-off rate of each unique set of search results.

Between pagination and the “Load more” approach, the latter was observed to have the added benefit of not replacing the first page of search results when results were loaded in, making it much easier for the participants to compare, for example, products from page 2 and page 1. Furthermore, the “Load more” approach enables the user to simply scroll back upward as the relevance of the results begin to decrease.

Loading a dynamic number of products

If you are looking to perfect the search results browsing experience with an advanced “Load more” implementation, you can consider having a dynamic number of products loaded in by default depending on the specific circumstances for each search query. For example:

  • If a user is searching within a scope that holds a limited amount of products, the search results list may encourage scrolling of the results (as the drop in result relevance can be presumed to be lower), and therefore include more results on “page 1” by default, compared to searches without a scope (which will often yield more results but typically of more varying relevance).
  • If a user is searching within a scope, the number of products loaded by default can also depend on the type of category searched within. For some categories, especially visually-driven ones, it can make sense for the user to explore a larger number of products. This is typically the case when the results can be easily reviewed by the user (i.e., load more products by default), which is often the case for visually-driven categories when compared to categories where multiple list item attributes must be assessed by the user to determine the item’s relevance (i.e., load fewer products by default), which tends to be the case for spec-driven categories.
  • If a user searches and the first 90 items have roughly the same relevancy score, the dynamic threshold for items loaded by default could increase, potentially all the way up to 90. Similarly, if results have a drastic drop in relevancy score after the 60th item, the default load threshold could be dynamically lowered. This would ensure that all highly relevant items are always displayed on the first page to encourage more focus and less browsing.

Guideline: Search results lists should use a “Load more” button for loading additional products, ideally with a dynamic number of products loaded by default.
